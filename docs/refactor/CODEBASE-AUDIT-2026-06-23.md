# Codebase deep audit — 2026-06-23

Produced by a 7-dimension multi-agent sweep (orphans-server, orphans-frontend, redundancy, deprecated, security, functional-gaps, tests-docs), each adversarially verified. **Analysis only — no code changed.** 47 raw findings → 28 prioritized items.

## Overall
Across 7 dimensions the verified findings collapse into a clear shape: the codebase is functionally healthy (no circular deps, redesign core flows well-tested, suite green) but carries (1) a small set of genuine security gaps, (2) a handful of real user-facing correctness bugs that survived the Phase-1/redesign teardown, (3) a large but low-risk layer of orphaned/deprecated cruft, and (4) widespread doc drift. The single most consequential theme is the delivery-fee model — three coexisting formulas plus a projection bug mean a claiming customer can be shown a fee that disagrees with what is actually charged (P1). The second is leftover service-area enforcement that contradicts the invite-only nationwide redesign and can hard-block out-of-state partners (P1). Security-wise, nothing is actively exploited, but the dependency tree has a reachable critical + axios SSRF highs (all non-breaking to fix), secrets silently fall back to dev defaults in prod, and the ADB-incompatible TTL means rate-limit windows can wedge — all cheap to fix and high value. The big judgment call the brief itself got wrong is the service-area cluster: it is NOT an orphan — it is mounted, frontend-wired, and tested, so it needs a coordinated decision + migration, not a delete. Recommended approach: ship the five quick security/correctness wins first (all S except the rate-limit fix), then run the fee unification as a careful move-then-delete sequence, then sweep orphans/redundancy/docs in batched one-concern PRs. Effort is front-loaded with small high-impact items; the only L items are the fee unification and the dependent deprecated-fields cleanup. Be honest with the owner on the open questions before any deletion in the service-area and order-status areas — those are the two places where 'dead-looking' code is actually load-bearing or externally reachable."
